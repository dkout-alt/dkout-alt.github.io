注解代替 XML 配置文件可以简化配置, 提高开发效率
原始注解主要是替代 的配置

常见注解

@Component 使用在类上用于实例化 Bean
@Controller 用在 Web 层类上用于实例化 Bean
@Service 用在 —– 层类上 —— ——————-

@AutoWired @Qualified @Resource 用于注入
@Value @Scope 注入普通属性 / 标注作用范围

@PostConstruct @PostDestroy 使用在方法上, 标注该方法时 Bean 的初始化/销毁方法

使用注解开发之前的 UserDao 和 Service 部分

使用注解创建 Bean 很方便 但我们无法通过注解得知其属于哪一层
因此有衍生注解：@Controller ， @Service @Repository

注解方式则可以不写

换句话说,按类型注入只需写autowired,按照名称写autowired和qualifier
@Resource 等于两者之和

ApplicationContext 的实现类

ClassPathXmlApplicationContext
从类的根路径下加载配置文件 ( 推荐 ) 也就是 resources 下
FileSystemXmlApplicationContext
从磁盘路径上加载配置文件, 配置文件可以在磁盘的任意位置
AnnotationConfigApplicationContext
这个呢 今天不能说的特别细
利用注解来加载 Spring 容器

getBean() 方法

getBean ( ) 方法的使用
我们拿出 getBean 的源码
public Object getBean (String name) throws BeansException {
//上面的 String name 就是在 xml 里配置的 bean 标签里的 id 属性
assertBeanFactoryActive();
return getBeanFactory().getBean(name);
}

public T getBean (Class requiredType) throws BeansException{
//传一个字节码对象类型
assertBeanFactoryActive();
return getBeanFactory().getBean(requiredType);
}
第一种方式允许容器当中出现多个相同类型的 bean , 因为 id 能够区分两个相同的类的对象 , 而第二种仅适用相反状况 , 更加简单 不需要强制转换

Spring 的配置文件

Bean标签基本配置
用于配置对象交由 Spring 来创建
默认情况下调用类中无参构造函数，若没有则不能创建

基本属性：id-唯一性标识-不允许重复，class-全限定名，默认反射通过无参构造来创建对象，需保证Bean内部必须存在无参构造

Scope 与生命周期

scope 指对象的作用范围 sigleton/prototype/request/session/global session 等

  • 通过创建test类确定singleton等的作用：容器中存在单一对象、多个对象
  • 探索Bean的创建时机
    若singleton创建容器时就已经被创建（加载 applicationContext 的时候
    实例化个数1个，Spring的核心文件加载时，实例化配置的Bean实例
    生命周期：
      对象创建：应用加载，创建容器时，对象被创建
  对象运行：只要容器存在，对象一直存活
  对象销毁：应用卸载，销毁容器时，对象被销毁
    若prototype，则在getBean的时候被创建，每getBean一次创建一次
    实例化个数：多个
    创建时机：-
    生命周期：
      当使用对象时，创建新的对象实例
  只要在使用中就存活
  长时间不用，会被 Java 垃圾回收机制回收

Bean生命周期的配置

init-method：指定类中的初始化方法名称
destroy-method 指定类中销毁方法名称

Bean实例化的三种方式

无参构造方法实例化（最重要）

工厂 + 静态方法
创建工厂，告诉Spring获得对象时，去用工厂进行返回——去改配置文件——改成去找对应的方法返回对象
